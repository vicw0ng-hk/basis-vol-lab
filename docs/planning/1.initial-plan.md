# Basis and Vol Lab Implementation Plan for a Hong Kong Quant Dev Transition

## Overview

Your proposed **Basis & Vol Lab** is the right kind of flagship project for a three-month transition, but it becomes much stronger if you tighten it into a **market-structure observability platform** rather than a broad “everything dashboard.” That narrower framing matches what current Hong Kong roles are signaling: Python plus the data stack, production-grade market-data handling, trader-facing analytics, performance awareness, cloud/containerization, and evidence that you can deliver reliable software around live markets. Current Hong Kong postings explicitly call for Python with Pandas, NumPy, and SciPy; quantitative-development experience in fast-paced fund or bank environments; Linux, TCP/UDP, and multithreaded systems knowledge; support for mission-critical trading software; AWS and Kubernetes familiarity; CI/CD discipline; and analytics pipelines for trading or finance teams. 

The key change I recommend is this: **launch with only BTC and ETH, and with only two venues doing the jobs they are best at**. Use **Deribit as the options venue** because its API natively exposes option IVs and Greeks plus perpetual funding and open interest in the ticker feed, and use **Binance as the futures/perps venue** because it exposes public funding history, basis, mark-price streams, and open-interest statistics. That gives you a coherent product question — “how are carry, funding, IV, skew, and OI lining up now, and how unusual is that regime?” — without paying for market data.

## What to build

The strongest launch product is a **cross-venue crypto-derivatives regime monitor** with four pages. The first page is a **live overview**: Deribit option-chain summaries, Deribit perp funding, Binance futures basis, Binance funding, and open-interest concentrations. The second page is **volatility and skew**: ATM term structure, selected smile nodes, and realized-versus-implied panels. The third page is **carry**: perp funding, quarterly basis, and carry/vol divergence. The fourth page is **historical replay**: percentile regimes and date-by-date “what changed?” summaries. This is ambitious enough to look like real desk tooling, but still small enough to finish in three months. The exchange APIs support this directly: Deribit’s ticker subscriptions expose IV, Greeks, funding, open interest, and related fields in real time; Binance exposes public mark-price/funding streams plus REST endpoints for funding history, basis, and open-interest statistics. 

To keep the project useful rather than generic, build a thin **signal layer** on top of raw metrics. I would define three original headline signals. A **carry-vol divergence score** should compare percentile-ranked annualized carry against percentile-ranked IV-minus-realized-vol spread. A **skew stress score** should combine front-end 25-delta risk-reversal extremes with concentration of option open interest by strike or expiry. A **regime-change alert** should fire when the one-day move in carry percentile, skew percentile, and OI concentration percentile crosses a fixed composite threshold. Those signals are straightforward to explain to an interviewer, they are non-magical, and they let you tell a trader-facing story without pretending to have discovered alpha.

For data sources, start with **Deribit public market data** and **Binance public derivatives data**, and do not add more venues until the product is already good. Deribit recommends WebSocket subscriptions instead of REST polling for real-time access, documents heartbeat handling for stale connections, and provides separate test and production environments with separate accounts and API keys. Binance documents hard connection lifetimes, ping/pong requirements, per-connection stream limits, and public endpoints for the basis/funding/OI pieces you need. Those are exactly the kinds of operational details that make your collector code look serious in an interview. 

You should also exploit the limited historical data that is already available on day one, so the dashboard does not look empty during your first week. Binance exposes public **basis** and **open-interest history** for the latest month, as well as public **funding-rate history**. Deribit exposes **historical volatility** and **volatility-index** chart data, alongside index-chart data. That gives you enough backfill to render first-pass historical panels immediately while your own collector is accumulating the richer history window over the next three months. 

## Local-first build order

The best way to finish this project on time is to treat the **MacBook build as the real product**, and treat cloud deployment as a later packaging exercise. The right local developer workflow is **uv for Python projects and dependencies**, with **uv workspaces** for a monorepo layout, and **mise** for non-Python tool versions and project tasks. uv’s current docs support workspaces, automatic locking and syncing, and a cross-platform `uv.lock` file; mise is designed to pin tools and define repeatable tasks in `mise.toml`. 

I would build in this order:

1. **Scaffold the monorepo and tooling first.** Create a uv workspace with `apps/web`, `apps/api`, and optional `apps/desktop`, plus `packages/connectors`, `packages/analytics`, `packages/contracts`, and `packages/persistence`. Put `uv.lock` under version control immediately, pin non-Python tools like `node`, `terraform`, and `wrangler` through `mise` (I will used brew-managed `aws-cli` locally), and define `mise` tasks for `lint`, `typecheck`, `test`, `collect`, `backfill`, `build:web`, and `deploy`. Use **Ruff plus a real type checker**; Ruff itself recommends pairing with a type checker, and Pyright is explicitly positioned as a high-performance static type checker for large Python codebases. 

2. **Build the local data model before any UI.** Use **SQLite locally for metadata** because Cloudflare D1 uses SQLite semantics; keep metadata tables limited to instruments, collection runs, alert definitions, and UI settings. Use **Parquet for time-series artifacts** and **DuckDB locally** for ad hoc analysis and development queries, because DuckDB can query Parquet efficiently and can read from S3-compatible object storage later as well. This local/cloud symmetry matters: it lets your MVP run entirely on your MacBook while preserving a clean migration path to Cloudflare D1 and R2. 

3. **Implement the Deribit connector next, not both connectors at once.** Start by discovering instruments, then subscribe only to the channels you actually need: tickers for active BTC and ETH options plus the relevant perps/futures. Deribit’s ticker subscription already gives you the crucial fields — bid/ask/mark IV, Greeks, open interest, mark price, and perpetual funding — and Deribit recommends subscriptions over REST polling for real-time data. Add heartbeat handling on day one and treat reconnect/resubscribe logic as part of the MVP, not polish. 

4. **Then add the Binance connector for basis, funding, and OI.** Use WebSocket mark-price streams for current funding and mark data, and REST backfills for funding history, basis, and open-interest history. Build the connector around Binance’s actual operational limits: WebSocket connections expire after 24 hours, require ping/pong handling, and allow a bounded number of streams and incoming messages per second. This is where your async experience becomes visible: `asyncio` is the right tool for network-bound collection, while Python’s own docs make clear that the GIL limits threads for CPU-bound work and that process pools are the correct escape hatch there. 

5. **Only after raw collection works should you build the analytics package.** Use **pandas** for normalization and time alignment, especially `resample` for bar construction; use **NumPy broadcasting** for vectorized pricing and batch transformations; use **SciPy** for IV root solving and interpolation across surface nodes; and serialize curated outputs as Parquet or compact JSON. The most interview-friendly validation task is to compute your own pricer outputs and compare them against Deribit’s published IV/Greek fields, then persist validation error summaries. That shows mathematics, numerics, quality control, and engineering discipline in one place. 

6. **Then build the product shell locally.** Keep the first web UI simple and fast: a static front end that reads versioned JSON artifacts and only calls dynamic endpoints for “refresh now” views. For local reproducibility and interview demonstration, containerize the web app and local API and wire them together with **Docker Compose**, which Docker documents as the YAML-driven way to define and run multi-container applications. Do not spend early time on animation or design polish; spend it on clear charts, fast loads, and reliable data freshness indicators. 

7. **Leave the optional desktop wrapper until late.** If, and only if, the core web product is already good, add a thin **PySide6 operator console** that consumes the same endpoints and surfaces watchlists, regime alerts, and recent charts. PySide6 is the official Qt for Python binding, so it is credible for desk-tooling and rapid-application-development box-ticking without pulling the whole project away from Python.

## Cloud rollout and cost controls

When you move to cloud, keep the architecture **Cloudflare-first for static delivery and storage, AWS-only for Python numerics and thin dynamic endpoints**. The reason is now very clear in the 2026 product docs. Cloudflare static asset requests are free and unlimited, but **Pages Functions are billed as Workers requests**, and the free Workers tier is constrained; D1 has explicit daily read/write limits and is designed around smaller SQLite-style databases; R2 gives you object storage with no egress fees and a usable free allowance; and AWS Lambda gives a meaningful free tier plus container-image support when Python scientific dependencies become awkward to package as ZIPs. 

The practical deployment sequence should be: first, deploy the **static web front end** to Cloudflare Pages; second, publish **historical and current artifacts** to R2; third, create a small **D1 database** only for metadata and run logs; fourth, expose a very small number of **Lambda endpoints** for on-demand analytics refresh or cache misses; and fifth, manage all of this with Terraform through HCP Terraform’s remote state, remote runs, and VCS integration. HCP Terraform’s current free entitlements still include those team-small workflow features, and the current feature-set docs show the familiar “free 500 managed resources” tier. 

Your biggest free-tier mistakes to avoid are straightforward. **Do not use D1 as a raw market-data sink** because the free plan only includes 100,000 written rows per day and 5 million read rows per day. **Do not use Workers for heavy analytics** because the free Workers tier remains limited. **Do not expose artifacts through `r2.dev` for your public demo**, because Cloudflare explicitly documents that `r2.dev` is for development traffic, is rate limited, and that production use should go through a custom domain. That implies a very clean storage rule: store **raw or semi-raw historical artifacts in R2**, **only metadata in D1**, and **only sparse dynamic computation in Lambda**. 

There is one more important operational nuance: **GitHub Actions cron is free and great for a public repo, but it is not a precise scheduler**. GitHub documents that standard GitHub-hosted runners are free and unlimited for public repositories, but scheduled workflows have a minimum frequency of five minutes and can be delayed or even dropped during periods of high load, especially at the top of the hour. That means you should not architect the whole product around “perfect every-minute cron.” Instead, use scheduled jobs for **historical snapshotting and daily compaction**, and use **manual refresh or cached on-demand endpoints** for the live page. 

The easiest way to stay within the R2 and D1 free tiers is to make your storage deliberately **curated rather than exhaustive**. Store **five- or fifteen-minute summary metrics** continuously, store **surface nodes** rather than the full smile at every strike for every timestamp, and store **full-chain snapshots only hourly or daily** for replay and validation. That design is not a compromise; it is actually better product design, because your regime explorer wants stable derived nodes and percentile histories more than it wants an unbounded tick archive. This is an inference from the current D1 and R2 limits, and it is the single biggest change I would make to your original plan. 

## Technical choices that maximize interview signal

The technical story you want interviewers to see is not “I built a crypto dashboard.” It is “I made disciplined engineering choices under realistic constraints.” Your concurrency story should be explicit: **`asyncio` for exchange I/O**, **NumPy/Pandas vectorization for bulk math**, and **`ProcessPoolExecutor` for CPU-heavy operations** such as calibration, replay batches, or benchmark comparisons. Python’s own docs support that split directly: `asyncio` is aimed at I/O-bound network code, threading is constrained by the GIL for CPU-bound work, and process pools side-step the GIL. 

Your analytics story should also be explicit. Use **pandas** to normalize timestamps and resample series, **NumPy** for vectorized calculations over arrays, **SciPy** for IV inversion and interpolation, **PyArrow/Parquet** for compact storage, and **DuckDB** locally for fast exploratory queries over Parquet files and later over S3-compatible storage. That stack is both technically strong and highly defensible in interviews because every piece is doing a job it is well suited for. 

I would also make validation a first-class feature, not a footnote. Deribit already gives you API-side IV and Greeks in real time, so your repo should include a **validation report** that compares your own calculations against those values and shows acceptable error bands by instrument type or tenor. This is precisely the kind of thing that separates a serious engineering project from an API visualization exercise.

Current Hong Kong roles care not only about development but also about testing, platform reliability, and trader-facing collaboration, so make those visible. Your README should open with the business question, then the architecture, then screenshots or a demo GIF, then exact local-run instructions, then the benchmark/validation section, and finally the deployment notes. On your CV, I would package the result in three evidence-driven bullets. First: **built a Python derivatives analytics platform** ingesting Deribit options and Binance futures/perpetual data and computing IV, skew, basis, funding, and regime metrics. Second: **implemented resilient async collectors** with reconnect, heartbeat, and rate-limit handling across public WebSocket and REST APIs. Third: **deployed a cost-aware serverless stack** using Cloudflare Pages, R2, and D1 plus AWS Lambda, provisioned with Terraform and automated through GitHub Actions. Those bullets align directly with the recurring Hong Kong skill signals around Python data libraries, real-time systems, cloud platforms, CI/CD, and market-facing tools. 

If schedule pressure hits, cut scope in this order: **desktop wrapper first, always-on streaming second, fancy UI third**. Do **not** cut the core analytics package, validation report, test suite, deployed demo, or benchmark notes. Those are the pieces that most directly answer the market signals from the roles you are targeting.  

**Open questions and limitations.** Two practical caveats remain. First, Cloudflare R2 is free to start but now requires enabling an R2 subscription through the checkout flow, so if you want to avoid any billing setup at the beginning, keep artifacts local during the initial MVP and add R2 later. Second, GitHub Actions scheduled runs are best-effort rather than hard-real-time, so your “live” experience should come from cached on-demand endpoints, not cron-only collection.
