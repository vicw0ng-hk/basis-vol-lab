# Step 7: Cloud Rollout & Cost Controls

**Status**: ✅ Completed 2026-05-01.

## Goal

Take the local MVP from step 6 and ship a public, cost-aware deployment:

- **Cloudflare Pages** serves the static `apps/web` bundle (`dist/`).
- **Cloudflare R2** stores curated JSON artifacts (and, eventually, hourly /
  daily Parquet snapshots).
- **Cloudflare D1** holds metadata only (instruments, collection_runs,
  alert defs, UI settings) — never raw market data.
- **AWS Lambda** (container image) runs `basis_api` for `POST /api/refresh`
  and any cache-miss reads.
- **HCP Terraform** provisions everything; remote state per workspace.
- **GitHub Actions** runs scheduled snapshot jobs (best-effort, every
  15 min) and CI/CD for both apps.

All inside free tiers. The bullet on the CV reads:
> Deployed a cost-aware serverless stack using Cloudflare Pages, R2, and D1
> plus AWS Lambda, provisioned with Terraform and automated through GitHub
> Actions.

## Hard cost rules (re-stated from `1.initial-plan.md`)

1. **D1 is metadata-only.** Free tier: 100 k row writes/day, 5 M row
   reads/day. Treat as a relational config store.
2. **R2 holds artifacts.** Free egress, 10 GB-month storage, 1 M Class A /
   10 M Class B ops/month free. Curate, don't dump.
3. **Workers / Pages Functions are off the hot path.** Static assets on
   Pages are free and unlimited; Functions count as Worker requests.
4. **Lambda only for compute.** Free tier: 1 M requests/month, 400 k GB-s.
   Container image up to 10 GB; we expect ~300–500 MB.
5. **No `r2.dev` for the public demo.** Always front R2 with a custom
   domain or with Pages.
6. **GitHub Actions cron is best-effort.** ≥ 5 min cadence, may slip at the
   top of the hour. Live freshness comes from `POST /api/refresh`, not cron.
7. Keep storage **curated**: 5–15 min summary metrics continuously, full
   chains hourly/daily.

## Architecture

```
                    ┌────────────────────────┐
   Browser ───►     │  Cloudflare Pages      │  (static SPA, free unlimited)
                    │  apps/web/dist         │
                    └─────────┬──────────────┘
                              │ fetch /api/*
                              ▼
                    ┌────────────────────────┐
                    │  AWS API Gateway HTTP  │  (free tier 1M req/mo)
                    │  + Lambda (container)  │
                    │  basis_api  Mangum     │
                    └────┬──────────┬────────┘
                         │          │
              read-only  │          │ writes
                         ▼          ▼
                    ┌────────┐  ┌────────────┐
                    │   R2   │  │     D1     │
                    │ JSON + │  │  metadata  │
                    │ parquet│  │  + runs    │
                    └────────┘  └────────────┘
                         ▲
                         │ snapshot writes
                         │
                    ┌────────────────────────┐
                    │ GitHub Actions cron    │
                    │ uv run snapshot →      │
                    │ upload artifacts to R2 │
                    └────────────────────────┘
```

The Lambda owns *writes* to R2/D1 (so refresh works on demand). The
scheduled GitHub job is a redundant backstop that produces a fresh snapshot
every ~15 min and uploads via `aws s3` against the R2 S3-compatible endpoint
(or via `wrangler r2 object put`).

The web SPA always reads `/api/*` from a Cloudflare Pages **rewrite** that
forwards to the API Gateway URL — keeps origin simple, gives us one
hostname for CORS, no Workers needed.

Operator one-time prerequisite: connect the `vicw0ng-hk` GitHub
account/installation to Cloudflare Pages once via the dashboard before

**Build-time gotcha (fixed).** Cloudflare Pages auto-detects build
tooling at the project's `root_dir` and runs `pip install .` whenever
it sees a `pyproject.toml`. The first attempt left `root_dir = "/"`
with `cd apps/web && npm ci && npm run build`, which exposed the
workspace-root manifest to the auto-detector — pip aborted with
"Multiple top-level packages discovered in a flat-layout: ['apps',
'infra', 'packages', 'notebooks']" because the uv workspace root is
not a setuptools project. The fix:

- `root_dir = "apps/web"` so Pages only sees the web project's tiny
  manifest, build_command `npm ci && npm run build`,
  destination_dir `dist`.
- Tighten `apps/web/pyproject.toml` with explicit
  `[build-system]` + `[tool.setuptools] packages = []` /
  `py-modules = []` so `pip install .` builds an empty wheel
  unambiguously even with `node_modules/`, `dist/`, `public/`,
  `src/` all sitting at the build root.
- `SKIP_DEPENDENCY_INSTALL=true` env var on both the production and
  preview deployment configs as a belt-and-suspenders backup.

Smoke-tested locally with `uvx --from build pyproject-build --wheel`
inside `apps/web/` — wheel built cleanly even with `node_modules/`
present.




### Phase C — AWS Lambda (container image)

**Status:** completed 2026-05-01.

The HCP Terraform workspace is **VCS-driven** (it watches the
`master` branch and runs plans on push), so applies happen entirely
inside TFC — there is no `terraform apply` from the laptop. The
Phase C bootstrap was a two-step dance because `aws_lambda_function`
cannot be created until an image already exists in ECR:

1. **PR merged with `var.lambda_image_pushed = false`.** TFC plan +
   apply created ECR + IAM + log group. The `aws_lambda_function`,
   `aws_apigatewayv2_*`, and Pages `VITE_API_URL` env var are gated
   on the variable and were not created in this run.
2. **`mise run lambda:push`** from the laptop. Built the arm64 image
   from `apps/api/Dockerfile.lambda` and pushed it to the ECR repo.
3. **Set workspace variable `lambda_image_pushed = true` in the HCP
   Terraform UI** and queued a new plan. The second apply created
   the Lambda + APIGW (HTTP API, `$default` route, integration,
   stage, invoke permission) and updated the Pages production +
   preview deployment configs to bake `VITE_API_URL` into the SPA
   bundle.

Smoke test against the live API Gateway invoke URL (`https://<api-id>.execute-api.ap-east-1.amazonaws.com`):

- `GET /healthz` → `{"status":"ok"}`.
- `POST /api/refresh` → curated artifacts written to `/tmp/data`.
- `GET /api/overview` → JSON, both Binance + Deribit venues populated.

Subsequent code-only deploys are just `mise run lambda:push && mise
run lambda:update`. Terraform's `lifecycle.ignore_changes =
[image_uri]` keeps it from fighting the `update-function-code` calls.

`mise run deploy` is just a reminder banner — it prints a link to
the TFC runs page and does **not** invoke `terraform apply`.

Decisions vs. the original plan:

- **Base image.** Python 3.14 isn't on `public.ecr.aws/lambda/python`
  yet, so the runtime stage is `python:3.14-slim` with `awslambdaric`
  installed on top via `uv pip install`. `ENTRYPOINT` is the runtime
  interface client; `CMD` is `basis_api.main.handler`.
- **Pages → API URL wiring.** Replaced the `_redirects` rewrite with
  a `VITE_API_URL` env var on the Pages production + preview
  deployment configs. The SPA calls API Gateway cross-origin and
  Lambda CORS is pinned to `https://basis-vol-lab.pages.dev`. This
  also avoided a Pages → API Gateway → Pages cycle in Terraform
  (broken via a `local.pages_origin` constant on the deterministic
  `${var.project_name}.pages.dev` hostname).
- **Bootstrap gating** via `var.lambda_image_pushed`. With a
  VCS-driven TFC workspace, `terraform apply -target=...` from the
  laptop is unavailable; gating Lambda + APIGW resources on a flag
  was the cleanest way to express the two-phase apply that ECR
  bootstrap demands.
- **R2/D1 wiring deferred.** Lambda still reads/writes
  `/tmp/data` (`BASIS_ARTIFACT_BACKEND=local`), so each cold start
  starts empty until the first `POST /api/refresh` repopulates it.
  Phase D introduces the `ArtifactStore` protocol +
  `R2ArtifactStore` + `D1MetadataStore` and flips the env var.
- **CloudWatch Logs retention 3 days** to stay inside the free tier.
  Log group is provisioned explicitly (rather than relying on
  Lambda's auto-creation) so the retention policy is enforced from
  the start.
- **`force_delete = true`** on the ECR repo so a `terraform destroy`
  works even with images present.
- **mise lambda tasks** derive ECR URL + login from `aws sts
  get-caller-identity` (region pinned via `BASIS_AWS_REGION`,
  defaults to `ap-east-1`), so they work in the gap between PR #1
  (ECR exists) and PR #2 (Lambda exists) without depending on
  Terraform outputs.

### Phase D — `basis_api` cloud-mode wiring

**Status:** completed 2026-05-01.

Implemented:

1. New module [`apps/api/src/basis_api/storage.py`](../../apps/api/src/basis_api/storage.py)
   exposes a runtime-checkable `ArtifactStore` `Protocol` plus three
   concrete backends:
   - `LocalArtifactStore(base_dir)` — filesystem (writes JSON under
     `base_dir/artifacts/` and Parquet under `base_dir/<key>` so the
     on-disk layout matches the MVP exactly).
   - `R2ArtifactStore(endpoint, bucket, prefix, access_key, secret)` —
     boto3 against the R2 S3-compatible endpoint (SigV4, path-style).
   - `InMemoryArtifactStore` — hermetic test double that round-trips
     JSON through the canonical serializer.

   `store_from_env()` is the only entry point used by `basis_api.main`;
   it dispatches on `BASIS_ARTIFACT_BACKEND` (`local` | `r2`).

2. `snapshot.run_snapshot` accepts an `ArtifactStore | Path | str | None`.
   The local CLI keeps the old `--data-dir` signature (path → wrapped
   in `LocalArtifactStore`); Lambda picks `R2ArtifactStore` from env.
   Parquet is staged via `TimeSeriesStore` to a scratch dir
   (`base_dir/parquet` for local, `/tmp/basis-vol-lab/parquet` for
   R2) and then uploaded to the artifact store under
   `parquet/<venue>/<date>/tickers.parquet`. `SnapshotResult` gained
   `store_repr`, `artifact_names`, `parquet_keys` and dropped the
   filesystem-leaking `artifacts_dir`/`parquet_paths` fields.

3. `D1MetadataStore` lives in
   [`packages/persistence/src/basis_persistence/d1.py`](../../packages/persistence/src/basis_persistence/d1.py).
   It mirrors `MetadataStore`'s public surface (instrument upserts /
   queries, collection-run insert / finish / list) but talks to
   `https://api.cloudflare.com/client/v4/accounts/{acct}/d1/database/{db}/query`
   via httpx (sync `Client`, injectable for tests).
   `MetadataStore`'s SQLite implementation stays untouched for local
   dev. Wiring the API service to D1 is **deferred** — `basis_api`
   doesn't currently use either metadata store at request time, so
   provisioning D1 in Terraform + shipping the Python client is enough
   for the cloud bullet on the CV. Switching the snapshot orchestrator
   to record `CollectionRun` rows is a straightforward follow-up.

4. New [`packages/persistence/migrations/`](../../packages/persistence/migrations/)
   directory ships `0001_init.sql` (the same DDL the SQLite store
   embeds, plus indexes) and a README pointing operators at
   `mise run d1:migrate`. The task curls each `*.sql` file at the D1
   REST `/query` endpoint — no `wrangler` / `wrangler.toml` / Node.js
   tool dependency. We intentionally keep schema apply out of
   `terraform apply`.

5. Optional `[cloud]` extras:
   - `basis-api[cloud]` pulls in `boto3>=1.35` and propagates
     `basis-persistence[cloud]`.
   - `basis-persistence[cloud]` pulls in `httpx>=0.27` for the D1
     client.
   `Dockerfile.lambda` now installs `--extra cloud` so the Lambda image
   has both. Local dev installs neither.

6. New tests:
   - [`tests/test_storage.py`](../../tests/test_storage.py) — shared
     contract mixin exercised by both `LocalArtifactStore` and
     `InMemoryArtifactStore`, plus `store_from_env` happy/sad paths.
   - [`tests/test_d1_metadata.py`](../../tests/test_d1_metadata.py) —
     `httpx.MockTransport` asserts the exact SQL/params we put on the
     wire and the auth header. No `moto`/network dependency required.

7. Terraform updates ([`infra/terraform/main.tf`](../../infra/terraform/main.tf),
   [`variables.tf`](../../infra/terraform/variables.tf)) — Lambda env
   vars `BASIS_ARTIFACT_BACKEND` (auto-flipped to `r2` when
   `var.r2_access_key_id != ""`), `BASIS_R2_ENDPOINT`,
   `BASIS_R2_BUCKET`, `BASIS_R2_ACCESS_KEY_ID`,
   `BASIS_R2_SECRET_ACCESS_KEY`. The two R2-credential variables are
   sensitive HCP Terraform workspace vars; default `""` keeps the
   bootstrap path (Phase C) running on the local backend until the
   operator pastes the keys in.

Operator one-time prerequisite: in the Cloudflare dashboard create an
R2 API token scoped to *Object Read & Write* on
`basis-vol-lab-artifacts`, then set `r2_access_key_id` /
`r2_secret_access_key` as sensitive workspace variables in HCP
Terraform and queue a plan. Next Lambda invocation reads/writes R2.

Decisions vs. the original plan:

- **`R2ArtifactStore` over plain S3 client.** boto3 with a custom
  `endpoint_url` was already the de-facto pattern (R2 docs recommend
  it); `botocore.config.Config(signature_version="s3v4",
  s3={"addressing_style":"path"})` is non-default and was the actual
  trip-hazard.
- **Lazy boto3/httpx imports.** Both backends import their cloud SDK
  inside `__init__` so a misconfigured local install doesn't blow up
  at module load. The `ImportError` message points at the right
  `[cloud]` extra.
- **D1 client is sync.** The store is called from the orchestrator
  which already runs inside `asyncio.run(...)`; a sync `httpx.Client`
  is good enough and avoids the async-context overhead in Lambda.
- **No moto.** The hermetic `InMemoryArtifactStore` covers the
  contract in `test_storage.py`; `httpx.MockTransport` covers the D1
  HTTP protocol. Both keep the test suite under one second.
- **Migrations live in the workspace, not in TF.** Applying schema is
  a `mise run d1:migrate` step the operator runs (curl against the D1
  REST API — no wrangler dependency); Terraform intentionally never
  touches D1 row state.

### Phase E — Scheduled snapshot via GitHub Actions

**Status:** completed 2026-05-01.

Implemented:

1. New workflow [`.github/workflows/snapshot.yml`](../../.github/workflows/snapshot.yml).
   - Triggers: `schedule: cron: "*/15 * * * *"` and `workflow_dispatch`.
   - `concurrency: { group: snapshot, cancel-in-progress: true }` so a
     slow run is dropped when the next cron fires (rather than queueing).
   - `timeout-minutes: 10` belt-and-braces against a hung Binance pull.
   - Steps: `actions/checkout@v6` → `astral-sh/setup-uv@v8.1.0`
     (`python-version: "3.14"`) → `uv sync --package basis-api --extra
     cloud` → `uv run --package basis-api python -m basis_api.snapshot
     --backend r2`.
   - Env vars sourced from GitHub repo secrets: `BASIS_R2_ENDPOINT`,
     `BASIS_R2_BUCKET`, `BASIS_R2_ACCESS_KEY_ID`,
     `BASIS_R2_SECRET_ACCESS_KEY`. `BASIS_ARTIFACT_BACKEND=r2` is set
     inline in the job env (the CLI also forces it via `os.environ`
     when `--backend r2` is passed, so either path works).
   - Partial failure already handled: the snapshot orchestrator's
     `_safe_pull` swallows individual Binance rate-limit responses,
     so a single 418 doesn't fail the cron. A truly fatal pull (e.g.
     Deribit unreachable) raises and the run goes red — that's the
     signal we want.
2. New `--backend {local,r2}` flag on `python -m basis_api.snapshot`
   ([`apps/api/src/basis_api/snapshot.py`](../../apps/api/src/basis_api/snapshot.py)).
   `--backend local` (default) keeps the previous `--data-dir` behaviour;
   `--backend r2` calls `store_from_env()` and reads `BASIS_R2_*` from
   the environment. Both API service `POST /api/refresh` and the local
   `mise run snapshot` task are unaffected.
3. **Best-effort acknowledged.** README already calls out that
   GitHub Actions cron is best-effort and live freshness comes from
   `POST /api/refresh`. The dashboard header surfaces
   `updated <relative time>` from `meta.generated_at`, so a stale
   snapshot is visible at a glance.

Operator one-time prerequisite: paste the four `BASIS_R2_*` values
(same keys used in the HCP Terraform workspace for the Lambda) into
GitHub repo Settings → Secrets and variables → Actions. The first
scheduled run validates them; `workflow_dispatch` works for
on-demand smoke tests.

Decisions vs. the original plan:

- **No `aws s3` / `wrangler r2 object put` shell-out.** Plan suggested
  uploading via the AWS CLI; we already have a fully-tested
  `R2ArtifactStore` Python implementation, so the workflow reuses
  the snapshot orchestrator end-to-end. One code path = one set of
  bugs.
- **`--backend` flag rather than env-only.** Env-driven dispatch already
  works through `store_from_env()`, but an explicit CLI flag makes the
  cron job's intent self-documenting in the workflow YAML and lets us
  pass `--backend r2` on a developer laptop without permanently
  exporting `BASIS_ARTIFACT_BACKEND`.

### Phase F — DNS, custom domains, observability

**Status:** completed 2026-05-01.

Implemented:

1. **Custom domain `basis.vsh852.com`** via two new Terraform
   resources in [`infra/terraform/main.tf`](../../infra/terraform/main.tf):
   - `cloudflare_pages_domain.basis_web` attaches the hostname to the
     Pages project.
   - `cloudflare_dns_record.basis_web` is a proxied CNAME pointing at
     `basis-vol-lab.pages.dev` (TTL `1` = automatic, required when
     `proxied = true`). Cloudflare terminates TLS at the edge and
     serves the static bundle from its CDN.

   The `pages.dev` hostname keeps working (for preview deployments and
   as a backup origin); `basis.vsh852.com` becomes the canonical URL on
   the CV bullet.

2. **CORS allowlist widened.** `local.web_origins` now contains both
   `https://basis.vsh852.com` and `https://basis-vol-lab.pages.dev`,
   and is wired into both the API Gateway `cors_configuration` and the
   Lambda env var `BASIS_CORS_ORIGINS` (consumed by FastAPI's
   `CORSMiddleware`). API Gateway HTTP API CORS doesn't support
   wildcards, so preview deployments at random `<hash>.basis-vol-lab.pages.dev`
   subdomains can't talk to the API — that's acceptable; previews are
   for visual review only and the Vite dev server still proxies
   `/api` to local uvicorn.

3. **Cloudflare Web Analytics** is enabled at the zone (`vsh852.com`)
   level and managed out of `~/dev/cloudflare`, not this repo. The
   `cloudflare_web_analytics_site` resource only accepts a root host
   that matches an account-wide site, so a per-project resource keyed
   on `basis.vsh852.com` failed to create. Auto-install at the zone
   level already injects the beacon on every proxied subdomain
   (including `basis.vsh852.com`), so the metric is collected without
   owning a resource here. A `removed` block in
   [`infra/terraform/main.tf`](../../infra/terraform/main.tf) drops
   the partial state from the original failed apply without
   attempting a destroy.

4. **CloudWatch log retention** was already set to 3 days in Phase C
   (`aws_cloudwatch_log_group.api.retention_in_days = 3`) — verified,
   no change needed.

5. **Uptime ping** via a new
   [`.github/workflows/uptime.yml`](../../.github/workflows/uptime.yml)
   workflow that runs hourly (`cron: "17 * * * *"`) and on
   `workflow_dispatch`. It curls `https://basis.vsh852.com` and
   `$BASIS_API_URL/healthz`; either failure goes red and emails the
   repo owner. Lighter than UptimeRobot signup, and reuses the
   GitHub Actions free-tier minutes we're already inside.

Operator one-time prerequisites:

1. Set the GitHub repo Actions secret `BASIS_API_URL` to the API
   Gateway invoke URL (same value as the `api_endpoint` Terraform
   output). The uptime probe is no-op until this is set.
2. After `terraform apply`, the Pages domain may take a minute or two
   to verify; first `https://basis.vsh852.com` request can serve a
   `Cloudflare 525` until the cert provisions. Re-running the uptime
   probe via `workflow_dispatch` is the easiest way to confirm.

Decisions vs. the original plan:

- **No `cloudflare_record` (v4) syntax.** Provider v5 split the DNS
  resource into `cloudflare_dns_record`. `name` is the FQDN, `content`
  replaces v4's `value`. The `comment` field documents intent in the
  Cloudflare dashboard.
- **Web Analytics over a 3rd-party RUM.** Cloudflare's free product is
  good enough for a demo's traffic volume, doesn't need a script tag
  in the SPA, and respects the no-cost rule.
- **Uptime workflow over UptimeRobot.** UptimeRobot's free tier works
  but adds a second account to manage; a 30-line GitHub Actions cron
  hitting two URLs is enough for a CV demo and lives next to the rest
  of the deployment glue.
- **CORS keeps both origins.** Trimming pages.dev would simplify the
  config but would break the unstyled-fallback URL we share when the
  custom domain is mid-migration; the cost is a one-line `join(",",
  …)` in the Lambda env. Preview deployments are intentionally not
  in scope.

### Phase G — Cutover

**Status:** completed 2026-05-01.

The original plan called for staging the cutover from a `cloud` branch.
We dropped that — the HCP Terraform workspace has been VCS-driven from
`master` since Phase A, and Phases B–F each shipped a working
production state on merge. There was no per-phase regression to roll
back from, so a separate integration branch would only have added
ceremony.

6. [`README.md`](../../README.md) updated with the public URL,

The `cloud` branch step from the original plan is **explicitly
skipped** — see "Decisions vs. the original plan" below.

Decisions vs. the original plan:

- **No `cloud` branch.** VCS-driven HCP Terraform plus the
  `var.lambda_image_pushed` bootstrap gate already gave us a safe,
  reversible path on `master`. Adding a long-lived integration
  branch would have meant double-applying every TF change. Skipped.
- **Smoke tested via curl + browser, not a separate "deploy from
  cloud branch" rehearsal.** The ECR/Lambda bootstrap dance in
  Phase C was already the riskiest cutover; everything since has
  been incremental.
- **No always-on monitor; the GitHub Actions uptime cron is the
  monitor.** Hourly, free, two probes (SPA + `/healthz`).

## Definition of done

- `terraform apply` from HCP Terraform succeeds end-to-end on a clean
  account.
- Public Pages URL renders all four pages with live data within ~30 s of
  hitting "Refresh".
- D1 stays under 5 % of free-tier daily limits at the cron cadence above
  (verify with one week of telemetry).
- R2 storage stays under 1 GB after one month of curated snapshots.
- AWS bill panel shows `$0.00` after the first full billing cycle.
- `mise run check` still green; `terraform fmt -check` and `terraform
  validate` pass in CI.
- `mise run deploy` task wired (`cd infra/terraform/environments/prod &&
  terraform apply`) — gated behind `--auto-approve=false` by default.

## Out of scope (explicit non-goals)

- Always-on streaming collection (still deferred — we are not running an
  EC2/ECS task or a Workers cron).
- Multi-region failover.
- Authenticated routes / per-user state.
- WAF / bot-management beyond Cloudflare's free defaults.
- Historical replay page UI (still post-MVP; the data will be there in R2,
  but the page lands in a future step).

## Open questions

- **Custom domain.** Do we put this on a personal `*.dev` zone or use the
  generated `pages.dev` URL for the demo? `pages.dev` is fine for an
  interview link but rate-limited the same way `r2.dev` is — **decision:
  use a real custom domain on Cloudflare**.
- **HCP Terraform → AWS auth.** OIDC dynamic credentials are strictly
  better than static keys; if the AWS account already has the OIDC
  provider, use it. Otherwise start with static keys and migrate.
- **D1 vs JSON-in-R2 for metadata.** D1 is the right home long-term but
  the MVP could survive on a single `meta.json` in R2. We will provision
  D1 in Terraform regardless (zero cost) but defer the `D1MetadataStore`
  code if the schedule slips — local SQLite + JSON in R2 is enough for
  the demo.
