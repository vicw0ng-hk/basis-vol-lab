# Step 7: Cloud Rollout & Cost Controls

**Status**: 🟡 Planning

## Goal

Take the local MVP from step 6 and ship a public, cost-aware deployment:

- **Cloudflare Pages** serves the static `apps/web` bundle (`dist/`).
- **Cloudflare R2** stores curated JSON artifacts (and, eventually, hourly /
  daily Parquet snapshots).
- **Cloudflare D1** holds metadata only (instruments, collection_runs,
  alert defs, UI settings) — never raw market data.
- **AWS Lambda** (container image) runs `basis_api` for `POST /api/refresh`
  and any cache-miss reads.
- **HCP Terraform** provisions everything; remote state per workspace.
- **GitHub Actions** runs scheduled snapshot jobs (best-effort, every
  15 min) and CI/CD for both apps.

All inside free tiers. The bullet on the CV reads:
> Deployed a cost-aware serverless stack using Cloudflare Pages, R2, and D1
> plus AWS Lambda, provisioned with Terraform and automated through GitHub
> Actions.

## Hard cost rules (re-stated from `1.initial-plan.md`)

1. **D1 is metadata-only.** Free tier: 100 k row writes/day, 5 M row
   reads/day. Treat as a relational config store.
2. **R2 holds artifacts.** Free egress, 10 GB-month storage, 1 M Class A /
   10 M Class B ops/month free. Curate, don't dump.
3. **Workers / Pages Functions are off the hot path.** Static assets on
   Pages are free and unlimited; Functions count as Worker requests.
4. **Lambda only for compute.** Free tier: 1 M requests/month, 400 k GB-s.
   Container image up to 10 GB; we expect ~300–500 MB.
5. **No `r2.dev` for the public demo.** Always front R2 with a custom
   domain or with Pages.
6. **GitHub Actions cron is best-effort.** ≥ 5 min cadence, may slip at the
   top of the hour. Live freshness comes from `POST /api/refresh`, not cron.
7. Keep storage **curated**: 5–15 min summary metrics continuously, full
   chains hourly/daily.

## Architecture

```
                    ┌────────────────────────┐
   Browser ───►     │  Cloudflare Pages      │  (static SPA, free unlimited)
                    │  apps/web/dist         │
                    └─────────┬──────────────┘
                              │ fetch /api/*
                              ▼
                    ┌────────────────────────┐
                    │  AWS API Gateway HTTP  │  (free tier 1M req/mo)
                    │  + Lambda (container)  │
                    │  basis_api  Mangum     │
                    └────┬──────────┬────────┘
                         │          │
              read-only  │          │ writes
                         ▼          ▼
                    ┌────────┐  ┌────────────┐
                    │   R2   │  │     D1     │
                    │ JSON + │  │  metadata  │
                    │ parquet│  │  + runs    │
                    └────────┘  └────────────┘
                         ▲
                         │ snapshot writes
                         │
                    ┌────────────────────────┐
                    │ GitHub Actions cron    │
                    │ uv run snapshot →      │
                    │ upload artifacts to R2 │
                    └────────────────────────┘
```

The Lambda owns *writes* to R2/D1 (so refresh works on demand). The
scheduled GitHub job is a redundant backstop that produces a fresh snapshot
every ~15 min and uploads via `aws s3` against the R2 S3-compatible endpoint
(or via `wrangler r2 object put`).

The web SPA always reads `/api/*` from a Cloudflare Pages **rewrite** that
forwards to the API Gateway URL — keeps origin simple, gives us one
hostname for CORS, no Workers needed.

## Step-by-step

### Phase A — Terraform skeleton ✅

**Status:** completed 2026-05-01.

1. **`infra/terraform/` layout — flat (deviation from initial plan).**

   ```
   infra/terraform/
     backend.tf      # cloud { organization = "vsh852", workspaces { name = "basis-vol-lab" } }
     providers.tf    # cloudflare + aws (OIDC default tags) + random
     variables.tf
     main.tf         # empty in Phase A; resources land in B–F
     outputs.tf      # empty in Phase A
     README.md
     .gitignore
   ```

   The plan originally proposed `environments/prod/` + per-resource
   `modules/`. We dropped both because we ship prod-only and the project
   fits comfortably in a single workspace — matches the user's existing
   `~/dev/aws` and `~/dev/cloudflare` flat style. If staging is ever
   added, copy the directory or use TFC workspace tagging. One HCP
   Terraform workspace (`basis-vol-lab`) drives remote state, remote
   plans, and VCS-triggered runs from `main` filtered to
   `infra/terraform/**`.

2. **Providers**
   - `cloudflare/cloudflare` (v5+) — Pages project, R2 bucket, D1 database,
     custom domain, DNS record.
   - `hashicorp/aws` (region `ap-northeast-1` Tokyo or `ap-southeast-1`
     Singapore — both outside Binance's restricted list and close to HK).
   - `hashicorp/random` for naming, `hashicorp/null` only if absolutely
     necessary.

3. **Secrets** (HCP Terraform workspace variables, marked sensitive):
   - `cloudflare_api_token` (scoped: Pages edit, R2 edit, D1 edit, Workers
     edit, DNS edit on the demo zone) — **terraform** category, sensitive.
   - `cloudflare_account_id`, `cloudflare_zone_id`, `domain` — **terraform**
     category, plain.
   - AWS auth via **OIDC dynamic credentials** (preferred over static
     keys), bootstrapped the same way as `~/dev/aws/bootstrap-oidc/`. Set
     workspace **env** vars `TFC_AWS_PROVIDER_AUTH=true` and
     `TFC_AWS_RUN_ROLE_ARN=<role arn>`.
   - `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY` for the S3-compatible
     endpoint (consumed by Lambda env + GitHub Actions, **not** by TF).

4. **CI guardrails**
   - `mise.toml` exposes `tf:fmt` (`terraform -chdir=infra/terraform fmt
     -recursive`) and `tf:validate` (`init -backend=false` + `validate`).
     `mise run check` now depends on `tf:fmt`.
   - `.github/workflows/terraform.yml` runs `fmt -check` + `init
     -backend=false` + `validate` on every PR touching `infra/terraform/**`.
     HCP Terraform owns plan/apply on merges to `main`.

### Phase B — Cloudflare resources

1. **R2 bucket** `basis-vol-lab-artifacts` with:
   - Public access disabled. Reads go through the Lambda or via signed
     URLs; we will *not* use `r2.dev`.
   - Lifecycle rule: delete `parquet/**` older than 30 days (re-evaluate
     once we know real volumes).
   - CORS rule allowing the Pages custom domain (only needed if we end up
     fetching directly from the SPA; default is no direct fetch).

2. **D1 database** `basis-vol-lab-meta` with the same schema
   `MetadataStore` already produces locally. Migration via
   `wrangler d1 migrations apply` from `packages/persistence/migrations/`
   (we'll generate these from the existing SQLite DDL — straightforward,
   D1 is SQLite).

3. **Pages project** `basis-vol-lab` with:
   - Build command `cd apps/web && npm ci && npm run build`
   - Build output `apps/web/dist`
   - Production branch `main`; preview deployments on PRs.
   - Custom domain (TBD — placeholder `basis.example.dev`).
   - **Pages Functions disabled.** All dynamic calls go to the Lambda via
     a `_redirects` rule:
     ```
     /api/*  https://<api-gw-id>.execute-api.<region>.amazonaws.com/api/:splat  200
     ```
     This is a static-asset rewrite (free), not a Function invocation.

4. **Vite config tweak** — add `VITE_API_BASE` env var so local dev still
   hits `http://localhost:8000` while production uses the same origin
   (relative `/api/...`). No code change beyond reading the env.

### Phase C — AWS Lambda (container image)

1. **ECR repo** `basis-vol-lab/api` (private, single region).

2. **Lambda function** `basis-vol-lab-api` packaged as the existing
   `apps/api/Dockerfile` adapted for Lambda. Two changes to the Dockerfile:
   - Use `public.ecr.aws/lambda/python:3.14` as the base for the runtime
     stage (or keep `python:3.14-slim` and add the AWS Lambda Runtime
     Interface Client + emulator manually — the public image is simpler).
   - `CMD ["basis_api.main.handler"]` — `Mangum` is already wired in
     `apps/api/src/basis_api/main.py`.

   Memory 1024 MB, timeout 30 s, ephemeral storage 1 GB. Architecture
   `arm64` — cheaper, Lambda Python 3.14 supports it.

3. **Environment**:
   - `BASIS_DATA_DIR=/tmp/data` (Lambda's writable scratch).
   - `BASIS_ARTIFACT_BACKEND=r2` — new env that switches `_load`/`run_snapshot`
     in `basis_api` to read/write S3-style URLs instead of the local
     filesystem.
   - `R2_ENDPOINT_URL`, `R2_BUCKET`, `R2_ACCESS_KEY_ID`,
     `R2_SECRET_ACCESS_KEY` — sourced from Lambda env (TF-managed, marked
     sensitive in HCP).
   - `BASIS_CORS_ORIGINS=https://basis.example.dev`.

4. **API Gateway HTTP API** (cheaper than REST API) routing `/{proxy+}` →
   Lambda. Default stage `$default`. Custom domain optional (Pages does
   the rewriting; clean URLs work either way).

5. **IAM** — least-privilege role: `logs:CreateLogStream/PutLogEvents` on
   its own log group, nothing else (R2 creds come via env, not IAM).

### Phase D — `basis_api` cloud-mode wiring

Code change scoped to `apps/api/src/basis_api/`:

1. New module `storage.py` with two backends:
   - `LocalArtifactStore(Path)` — current behaviour.
   - `R2ArtifactStore(endpoint, bucket, prefix, access_key, secret)` —
     uses `boto3` (already a small dep) against the R2 S3 endpoint.
   Both expose `read_json(name) -> dict`, `write_json(name, payload)`,
   `write_parquet(path, bytes)`.

2. `snapshot.run_snapshot` accepts an `ArtifactStore` instead of a `Path`.
   Local dev keeps the existing CLI signature (defaults to
   `LocalArtifactStore("data")`); Lambda picks `R2ArtifactStore` from env.

3. `MetadataStore` gains a tiny `D1MetadataStore` sibling that talks to
   the D1 HTTP API (Cloudflare's REST endpoint — `boto3` not applicable).
   We keep the SQLite implementation for local dev.

4. Add `basis-api[cloud]` extra in `apps/api/pyproject.toml` pulling in
   `boto3` and `httpx` (latter already present). Production image installs
   the extra; local dev does not have to.

5. Tests: an in-memory `InMemoryArtifactStore` fake plus contract tests
   that both `LocalArtifactStore` and `R2ArtifactStore` (via `moto`)
   satisfy the same protocol. Adds maybe 50 lines.

### Phase E — Scheduled snapshot via GitHub Actions

1. New workflow `.github/workflows/snapshot.yml`:
   - Triggers: `schedule: cron: "*/15 * * * *"` and `workflow_dispatch`.
   - Steps: `uv sync --package basis-api`, run
     `python -m basis_api.snapshot --backend r2`, exit cleanly on partial
     failure (the snapshot orchestrator already has `_safe_pull`).
   - Secrets: `R2_*` and `BASIS_*` from the GitHub repo secrets store.
   - Concurrency group `snapshot` with `cancel-in-progress: true` so
     overlapping runs don't pile up.
2. **Best-effort acknowledged.** The README and the dashboard's "freshness"
   stat both make clear that liveness comes from the manual refresh.

### Phase F — DNS, custom domains, observability

1. DNS: one CNAME for the Pages site, one for the API (optional). Owned in
   the same Cloudflare zone, managed by Terraform.
2. Cloudflare Web Analytics on the Pages project (free, no JS budget hit).
3. AWS CloudWatch log retention set to **3 days** to stay inside free
   tier ingestion.
4. Tiny `/healthz` ping from a free uptime monitor (UptimeRobot's free
   tier or a cron in GitHub Actions hitting the URL once an hour).

### Phase G — Cutover

1. Deploy Pages from a `cloud` branch first; verify with a preview URL.
2. Deploy Lambda + API GW from HCP Terraform; smoke `curl
   https://api…/healthz`.
3. Run `POST /api/refresh` once; confirm artifacts land in R2 and the
   SPA renders against them.
4. Flip the Pages custom domain to `main` branch; merge `cloud` → `main`.
5. Update README with the public URL and the architecture diagram.
6. Update `docs/progress.md`: step 7 → completed, no more deferred items
   blocking the CV bullets.

## Definition of done

- `terraform apply` from HCP Terraform succeeds end-to-end on a clean
  account.
- Public Pages URL renders all four pages with live data within ~30 s of
  hitting "Refresh".
- D1 stays under 5 % of free-tier daily limits at the cron cadence above
  (verify with one week of telemetry).
- R2 storage stays under 1 GB after one month of curated snapshots.
- AWS bill panel shows `$0.00` after the first full billing cycle.
- `mise run check` still green; `terraform fmt -check` and `terraform
  validate` pass in CI.
- `mise run deploy` task wired (`cd infra/terraform/environments/prod &&
  terraform apply`) — gated behind `--auto-approve=false` by default.

## Out of scope (explicit non-goals)

- Always-on streaming collection (still deferred — we are not running an
  EC2/ECS task or a Workers cron).
- Multi-region failover.
- Authenticated routes / per-user state.
- WAF / bot-management beyond Cloudflare's free defaults.
- Historical replay page UI (still post-MVP; the data will be there in R2,
  but the page lands in a future step).

## Open questions

- **Custom domain.** Do we put this on a personal `*.dev` zone or use the
  generated `pages.dev` URL for the demo? `pages.dev` is fine for an
  interview link but rate-limited the same way `r2.dev` is — **decision:
  use a real custom domain on Cloudflare**.
- **HCP Terraform → AWS auth.** OIDC dynamic credentials are strictly
  better than static keys; if the AWS account already has the OIDC
  provider, use it. Otherwise start with static keys and migrate.
- **D1 vs JSON-in-R2 for metadata.** D1 is the right home long-term but
  the MVP could survive on a single `meta.json` in R2. We will provision
  D1 in Terraform regardless (zero cost) but defer the `D1MetadataStore`
  code if the schedule slips — local SQLite + JSON in R2 is enough for
  the demo.
